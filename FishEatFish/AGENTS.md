## 美术与 UI

- 正式资源必须通过工作区 `.agents/skills/jimeng-ai` 接口生成，或使用已登记且授权明确的外部位图资源。
- 每次调用美术 AI 前，必须向用户展示将被原样使用的完整提示词、提示词版本、模型、尺寸、数量、用途和输出目录，并等待明确确认。
- 用户确认后不得静默修改提示词；修改提示词或重新生成必须创建新版本、写入 `docs/art-prompts.md` 并重新确认。
- 原图、加工步骤、选定结果、授权状态、导入路径和 `.meta` 必须登记在 `docs/art-assets-register.md`。
- 鱼类角色的所有动画提示词和资源登记必须明确“原图朝左”或“原图朝右”；同一角色默认必须统一朝向。导入资源与提示词方向不一致时，必须在鱼配置的 `animationArtFacingDirections` 中记录实际方向，由加载层归一化，禁止在移动、攻击或远端同步代码中分别写死正负缩放。
- Sprite Sheet 导入前必须检查每个角色主体的 Alpha 包围盒和画格边缘；角色、触须、尾鳍或特效跨越固定网格时，禁止直接按等宽画格硬切。无须重生成即可修复时，应从扩展画格提取本帧最大连通主体并统一比例、锚点与透明安全边距；原图确实粘连或主体缺失时，必须建立新提示词版本并再次获得用户确认。
- 服务器快照应用必须保持幂等：`appearanceId`、动作或配置值未变化时，禁止重新装载同一动画资源、重置帧索引、累计时间或当前动作。周期性快照只能校正权威状态，不能让待机或技能动画反复从第 0 帧开始。
- 鱼儿逻辑方向固定为 `0°=向右、180°=向左`；美术默认方向只读取鱼配置 `artFacingDirection`。以后方向异常优先修改配置，不得修改服务端命中角度或散落修改节点翻转代码。
- API 密钥、生成临时文件和未选定候选图不得提交版本库。

## UI 操作前强制门禁

- 任何涉及 Canvas、Camera、HUD、安全区、Widget、按钮、摇杆、血条、用户名、弹窗、登录框、`EditBox`、触摸区域或世界坐标转 UI 坐标的任务，开始修改前必须完整读取本文件和 `docs/ui-layout.md`。
- 读取后必须先在对话中明确说明：目标 UI 的当前父节点链、使用的坐标系、设计分辨率、相机关系、锚点/Widget 规则，以及需要覆盖的 IDE 与浏览器运行环境；完成这些检查前禁止编辑 UI 代码或场景。
- 修改前必须检查 `assets/scenes/MainScene.scene` 和现有 UI 创建代码，不得根据截图猜测 Canvas 原点、节点锚点或相机补偿值。
- UI 修复必须先写清根因，再修改最小范围；不得用反复增加固定偏移、复制另一套布局或移动父节点来试错。
- 每次 UI 缺陷修复后，必须把新发现的根因和防复发规则追加到本节或 `docs/ui-layout.md`，不得让同一问题重复出现。

## UI 层级与坐标规则

- 世界对象只能放在 `WorldRoot` 体系；固定屏幕 UI 只能放在 `HudRoot/SafeAreaRoot/InputLayer` 体系。禁止把摇杆、技能按钮、登录弹窗或固定 HUD 放入地图世界节点。
- 当前场景的设计分辨率为 1280×720，Canvas 使用中心锚点；`WorldRoot` 与 `HudRoot` 都是 Canvas 的直接子节点，二者局部原点必须保持 `(0,0)`，由 Canvas 自身的屏幕对齐负责把中心放到正确世界位置。禁止因为 Canvas 在场景数据中显示为 `(640,360)`，就再次给子根节点减去 `640/360`。
- 游戏世界坐标使用 `WorldRoot` 的中心原点，鱼儿、地图和 `MainCamera` 都使用该分支的局部坐标；固定 HUD 使用 `HudRoot` 的中心原点。镜头移动到 `(cameraX,cameraY)` 时，`HudRoot` 只同步相同局部 XY，不得写成 `(cameraX-640,cameraY-360)`。
- Cocos Scene 编辑视图、游戏 `MainCamera`、Canvas/HUD 局部坐标和浏览器 DOM 是四套不同坐标系统：Scene 编辑视图由编辑器观察相机控制；游戏画面由 `MainCamera` 控制；Canvas/HUD 采用中心原点且 Y 轴向上；DOM 采用视口左上原点且 Y 轴向下。任何转换必须明确源和目标，禁止直接复用另一坐标系的位置值。
- IDE 画面出现三轴移动手柄、白色相机取景框或透视网格时，说明当前看到的是 Scene 编辑视图而非纯 Game View；不能把编辑器观察相机的透视投影当作实际游戏 UI 布局结果。但 Canvas、WorldRoot、HudRoot 与 MainCamera 的中心仍必须在场景中对齐，禁止依靠运行时相互抵消的错误根偏移让 Web 结果“碰巧正常”。
- 当前工程的 `MainCamera` 会移动，`HudRoot` 通过相机位置补偿保持屏幕固定。修改 `HudRoot`、Canvas 父子关系或 `followPlayer()` 前必须评估全部 HUD、弹窗、血条和输入控件，禁止只修一个控件。
- 禁止在未计算 Canvas `UITransform` 锚点、Canvas 节点位置、相机位置和父节点世界矩阵时，直接把屏幕 UI 改挂 Canvas 并写死 `±640/±360`。截图中登录面板被裁到左下角就是此类错误。
- 摇杆必须是一个完整复合组件：底盘、操纵帽、显示、触摸区域放在同一个 `JoystickControlRoot`，整体左下对齐；不得让显示位置和事件触发区域分离。
- 普通攻击与技能按钮必须放在同一个 `ActionControlsRoot`，整体右下对齐；按钮内部只使用容器局部坐标，不得分别计算屏幕位置。
- 右上功能入口必须先组成完整 `TopRightFeatureRoot`，再由该整体通过 Widget 固定在 `InputLayer` 右上角；`SkillLoadoutEntryRoot` 与 `TransformEntryRoot` 在容器内部左右排列，各自图标、文字和触摸区不得分离。技能配置与变身弹窗必须与入口共用 HUD 坐标语义并阻断下层输入，不得挂入 `WorldRoot`。
- 血条和用户名属于独立 UI Overlay，使用世界坐标到 UI 坐标转换跟随鱼儿；不得作为鱼儿子节点旋转、缩放或翻转。
- 一个布局链只能使用一种明确的坐标语义。不得在同一控件中混用世界坐标、Canvas 局部坐标、HUD 补偿坐标和浏览器 DOM 坐标。
- 使用 `Widget` 时先对齐整体容器，再布置内部节点。禁止同时用 `Widget` 和每帧 `setPosition()` 控制同一节点的同一轴。
- 正式显示和触摸命中必须属于同一组件根节点；修改尺寸、锚点或缩放后必须同时验证显示矩形与点击矩形。

## EditBox 与 IDE 输入规则

- Cocos Web/IDE 中的 `cc.EditBox` 点击后会在 `game.container` 下创建原生 HTML 输入元素；该元素不是场景树子节点，不能假设它天然继承 Cocos 父节点的裁剪、相机补偿和 IDE Game View 缩放。
- 用户名等单行输入必须在组件启用前显式设置 `EditBox.InputMode.SINGLE_LINE`。禁止使用默认 `InputMode.ANY`，因为它会创建带滚动条的 `<textarea class="cocosEditBox">`。
- 输入框背景与 `EditBox` 可使用父子节点组合，但必须共享同一个稳定屏幕空间容器。禁止把 `Graphics` 与 `Sprite` 加到同一节点；二者都是渲染组件，会产生组件冲突。
- 字体颜色、占位符颜色、背景颜色、字号、行高和输入模式必须显式设置，不得依赖平台默认样式，避免白底白字或 IDE/浏览器样式差异。
- 如需修正 `cocosEditBox` 原生 DOM 位置，必须依据 `GameCanvas.getBoundingClientRect()`、实际容器矩形、设计分辨率和设备像素比计算，并处理聚焦、失焦、窗口缩放与销毁清理；禁止只写固定 `left/top` 或猜测 DPR。
- IDE 与浏览器允许存在薄适配层，但不得维护两套互不关联的面板结构和布局常量。布局尺寸、锚点、输入校验和提交逻辑必须共用定义。
- 模态登录框显示期间必须暂停玩家移动和相机跟随，并阻断其下方摇杆、技能按钮与地图输入。

## UI 验收规则

- UI 修改完成后必须分别验证 Cocos IDE 内嵌预览和 Web 浏览器预览；浏览器正常不能代替 IDE 验收，IDE 正常也不能代替浏览器验收。
- 登录输入框至少验证：未聚焦位置、点击后位置、输入文字、中文输入法、失焦、再次聚焦、提交按钮，以及预览窗口缩放后的对齐。
- 摇杆和技能 UI 至少验证：显示位置、实际触摸位置、按下/移动/松开、地图移动时固定屏幕、不同宽高比和安全区。
- 血条和用户名至少验证：鱼儿移动、左右翻转、攻击、死亡、复活和镜头移动时始终保持屏幕正向并正确跟随。
- 自动测试、类型检查和卫生检查只能证明结构与代码基线，不能证明视觉布局正确。未经过用户在目标 IDE/浏览器实际预览确认时，结果必须标记为“待用户视觉验收”，不得宣称已经完全修复。
- 每次 UI 修改后必须检查控制台是否存在组件冲突、Promise rejection、资源加载和坐标相关错误，并保留用户原有正常平台行为。

## 已发生 UI 故障的防复发结论

- 摇杆显示与触摸区域分离：根因是显示节点和事件节点使用不同容器与坐标；以后必须使用单一复合根节点。
- 摇杆和技能跟随地图或位置混乱：根因是固定 UI 进入世界坐标或按钮分别对齐；以后固定放入 HUD 安全区并按整体容器对齐。
- 血条、用户名随鱼旋转：根因是作为世界角色子节点；以后使用独立 UI Overlay 和世界到 UI 坐标转换。
- `Graphics` 与 `Sprite` 组件冲突：根因是同一节点添加两个派生渲染组件；以后背景与输入/图片拆成父子节点。
- 输入框白底白字：根因是依赖默认字体样式；以后显式设置输入文字和占位符颜色。
- IDE 中 `textarea.cocosEditBox` 跑到右上角：根因是原生 DOM 与 Cocos/IDE 内嵌预览坐标、DPR 和相机补偿不一致；以后单行输入禁用默认 textarea，并按实际 Canvas 矩形验证 DOM 定位。
- 登录遮罩和面板被裁到左下角：根因是把 HUD 弹窗改挂 Canvas 后仍使用错误的 Canvas 原点补偿；以后禁止未经世界矩阵验证改变屏幕 UI 父节点。
- IDE Scene 视图中 Canvas 取景框与 HUD 平面分离：根因是 Canvas 已完成中心定位后，`WorldRoot` 和 `HudRoot` 又重复减去半屏尺寸；浏览器中相机与 HUD 同时错位后相互抵消，掩盖了场景架构错误。以后 Canvas 直系世界/HUD 根节点固定使用局部 `(0,0)`，相机与 HUD 只同步同一局部移动量。

## 架构与平台

- 核心玩法层不得导入 `cc` 或任何平台 SDK。
- 微信、抖音、Android、iOS、HarmonyOS API 只能出现在各自平台适配器内。
- UI 通过视图状态或命令接口连接玩法，不直接修改核心规则。
- 玩家标准图、游动、攻击和受击动作资源必须由 `AnimationsResManager` 统一加载、缓存、方向归一和按形态查询；`GameBootstrap` 只协调动作状态、播放时机、网络事件与角色节点，禁止重新维护动作资源路径、形态资源映射或成组帧数组。
- Cocos 资源文件与对应 `.meta` 必须一起提交，不得手工复用其他资源的 UUID。
