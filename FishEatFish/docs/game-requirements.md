# 《鲫鱼吃鲤鱼》游戏需求基线

需求版本：REQ-0.2
状态：已确认  
基线日期：2026-07-13

## 产品定位

游戏是一款横屏 2D 海洋大世界单机 PvE 游戏。玩家初始控制一条小鲫鱼，在有限但较大的海域中探索，与 AI 鱼类相互攻击并成长。

## 核心玩法

- 玩家使用虚拟摇杆进行二维游动。
- 初始普通攻击为面向当前朝向的短距离撕咬。
- 初始技能为带冷却的“鲫鱼抢食”（原冲刺撕咬）。
- 不同技能对应不同撕咬动作标识，供后续动画系统绑定。

## 工具复用

- 通用图片处理能力必须放置在工作区 `Tools/image-processing`，不得写死《鲫鱼吃鲤鱼》的目录、资源名称或生成文件名。
- 项目内只保留项目配置、资源登记和明确的导入参数；通用工具不得直接修改项目资源，除非调用时明确传入项目内输出路径。
- 击败其他鱼后获得经验；升级提高生命上限并回复部分生命。
- 其他鱼由 AI 控制，可攻击玩家或其他鱼。

## 世界与镜头

- 首版采用有限大海域，后续可升级为分区流式加载。
- 使用位于水中的 2D 正交镜头，从海底正上方垂直向下俯视；地图不得出现水面、天空、地平线或侧视海底剖面。
- 镜头跟随玩家在海底平面内移动。
- 鱼身体使用圆形或胶囊形轻量命中判定，核心结算不依赖完整刚体模拟。

## UI 与输入

- 横屏设计分辨率为 1280 × 720。
- 左侧预留虚拟摇杆，右侧预留普通攻击和初始技能按钮。
- UI 必须适配安全区、刘海屏、平板和折叠屏。
- 正式 UI 和美术禁止使用 SVG 或程序化图形拼凑。
- 所有 UI 修改必须遵守项目 `AGENTS.md` 的 UI 操作前门禁、坐标分层、EditBox 和 IDE/Web 双端验收规则；自动测试不得替代目标运行环境的视觉与交互验收。
- Canvas、WorldRoot、HudRoot 与游戏相机必须采用统一中心基准；不得通过世界根和 HUD 根同时增加相反固定偏移来让单一运行环境的结果碰巧对齐。

## 平台范围

- 微信小游戏
- 抖音小游戏
- Android
- iOS
- HarmonyOS

本阶段不接入平台 SDK，只建立统一接口和构建入口。正式导出仍需要各平台 SDK、AppID、签名和工具链；iOS 原生工程需要 macOS。

## 性能基线

- 主流设备目标 60 FPS，中低端设备不得低于 30 FPS。
- 后续实现必须使用对象池、视野外休眠和分区激活控制鱼群成本。
- 正式资源应使用图集和平台适配纹理，小游戏资源需支持首包与远程包拆分。
- 第一版性能预算暂定：最多 30 条活动鱼、屏幕内最多 16 条完整更新鱼；实际数值须通过真机压测修订。

## 本阶段范围

包含工程骨架、场景层级、核心类型、配置样例、存档契约、平台接口、构建入口、测试、需求和美术档案。

不包含多人战斗结算、商业 SDK、签名安装包和生产数据库；多人可见与同步移动由 REQ-0.5 覆盖。

## REQ-0.5 多人联机移动

- 服务端采用 Node.js 20、TypeScript、Fastify 和 WebSocket。
- 默认海域匹配后进入单房间，最多 50 名玩家。
- 客户端发送移动输入，服务器以 20Hz 校验位置和旋转并以 10Hz 广播快照。
- 客户端支持本地预测和服务器校正，远端玩家使用快照插值显示。
- 测试版使用内存账号和房间；断线玩家立即退出房间。
- 本阶段不结算普通撕咬、“鲫鱼抢食”、伤害、死亡或复活。

## REQ-0.2 初始化地图与玩家移动

- `MainScene` 显示首张正式海洋地图背景和玩家小鲫鱼。
- 玩家小鲫鱼使用 6 帧透明 PNG 循环游泳动画。
- 桌面预览支持 WASD 和方向键；移动端及小游戏支持屏幕左侧拖动式虚拟摇杆输入。
- 玩家可以在 3840 × 2160 的首张海域内自由游动，不得越过地图边界。
- 玩家仍可向平面内任意方向移动，但鱼儿美术朝向仅保留左、右两种；上下移动和停止时维持最近一次水平朝向，不允许鱼身随移动方向上下或斜向旋转。
- 鱼儿逻辑方向固定为 `0°=向右、180°=向左`；游泳、攻击等原图实际方向由鱼配置登记，并在加载时归一到 `artFacingDirection`。方向错误只能通过配置修正，不得分别修改移动、攻击或联机同步逻辑。
- 镜头平滑跟随玩家，并在地图边缘停止继续移动。
- 本阶段的虚拟摇杆只实现交互区域和文字提示；正式摇杆 UI 需另行确认美术提示词后生成。

## REQ-0.3 地图视角修订

- 首张地图改为水中垂直向下俯视海底的平面 2D 视角。
- 海底保持开阔，礁石、珊瑚和水草主要分布于边缘，中央区域供玩家自由游动。
- 禁止水面、天空、地平线、侧视海底剖面和斜俯视透视构图。
## REQ-0.4 UI 操作骨架

- 横屏安全区提供左下虚拟摇杆，支持拖动方向并驱动玩家移动。
- 下方偏中间提供普通撕咬按钮，右侧提供“鲫鱼抢食”技能按钮。
- 普通撕咬具备 0.55 秒冷却并生成朝向扇形攻击反馈；“鲫鱼抢食”具备 5 秒冷却，沿鱼儿朝向突进 240 世界单位并生成冲刺/抢食反馈。伤害、命中和多人战斗结算仍由后续 CombatSystem 接入。

## REQ-0.7 受击动作表现

- 服务器确认命中后，主动攻击鱼播放现有撕咬、咬合和甩头撕扯动画，被攻击鱼同步切换为独立受击序列帧。
- 受击表现只替换美术帧，不修改被攻击鱼的世界位置、服务器位置、移动状态或网络快照坐标，不增加击退、吸附、抓取或控制效果。
- 受击动画必须与攻击动画的咬合节奏对应，通过鱼体局部受力弯曲、表情、鳍尾惯性和上下甩动姿态表现被咬住与甩扯；每帧角色中心、锚点、尺寸和画格位置保持一致。
- 咬合与甩扯的核心帧中，受击鱼应翻转为白色肚皮朝向镜头上方的“翻肚”姿态，再在松开后恢复正常游姿；全程保持身体完整，不出现血液、伤口、裂口或恐怖表现。
- 本地玩家和远端玩家收到同一服务器伤害事件后播放相同受击动画；死亡时由受击动画衔接死亡表现。

## REQ-0.8 鱼儿头顶生命条 UI

- 每名本地与远端玩家鱼儿都必须在头顶中上方显示生命条，生命条属于独立 HUD Overlay，不得作为鱼节点子节点。
- 生命条位置必须由鱼儿世界坐标转换为 `FishHealthOverlay` 局部坐标，并在鱼儿移动、镜头移动、左右翻转、攻击、受击、死亡和复活时持续跟随且保持屏幕正向。
- 生命条使用正式位图边框、空槽与可水平裁剪的生命填充；禁止使用 SVG、`Graphics` 或纯色临时块拼凑正式效果。
- 生命填充比例使用 `health / maxHealth`，从左向右显示剩余生命，扣血时右端随比例缩短，生命为零时填充完全消失。
- 当前生命值与最大生命值以 `当前/最大` 的数字格式居中叠放在血条最前层，数字不得被边框或填充遮挡。
- 所有玩家复用同一血条组件结构和更新逻辑；服务器状态是多人生命值的最终来源，本地单机模式保留安全默认值。
- `FishHealthBarOverlay` 独立负责生命条、生命数字与 120 px 跟随偏移；`FishNameOverlay` 独立负责用户名与 160 px 跟随偏移。`GameBootstrap` 只负责实例生命周期及统一投影更新，不得直接操作组件内部 Sprite 或 Label。

## REQ-0.9 参数化登录对话框

- 登录对话框必须由独立 `LoginDialog` 组件创建；调用方通过 `variant`、标题、说明、输入占位符、按钮文本、初始用户名、长度限制和提交回调参数区分不同登录入口。
- Cocos 形态固定创建在 `HudRoot/SafeAreaRoot/InputLayer` 的 1280×720 HUD 中心坐标；Web 形态复用同一套内容与校验参数创建 DOM 输入框，IDE 预览使用单行 `EditBox` 与 GameCanvas 矩形定位。

## REQ-0.10 玩家对象与角色生命周期

- 所有可见玩家必须使用 `Player` 对象，封装共用的节点、精灵、位置、左右朝向、生命、死亡、缩放、透明度和帧切换逻辑；远端玩家只使用这些通用能力。
- 本地客户端主角必须使用 `LocalPlayer extends Player`，在共用能力上封装输入驱动的预测移动和转向；仅本地独有的输入、网络发送与镜头跟随由本地流程调用。
- `RoleManager` 统一创建、查询和移除 `Player/LocalPlayer`；不得在 `GameBootstrap` 或网络注册表中直接创建、销毁玩家 Node。
- IDE 内嵌预览与 Web 浏览器预览共用同一套 Cocos `Sprite + Label` 实现，并分别进行视觉验收。

## REQ-0.9 多人动作同步

- 普通撕咬和“鲫鱼抢食”由服务器生成递增动作序号，并在 `skillEffect` 与后续状态快照中携带当前动作。
- 每个客户端对同一动作序号只播放一次；远端鱼即使错过瞬时技能事件，也能从状态快照补偿播放对应动作。
- 远端动作只替换角色 SpriteFrame，不修改远端鱼的服务器位置、世界坐标或朝向规则。

## REQ-0.10 远端鱼儿待机动画

- 每名远端鱼儿创建后必须循环播放游泳待机序列帧，不能停留在单张初始帧。
- 远端撕咬、受击和死亡动作期间暂停待机循环；动作结束或复活后恢复游泳循环。
- 待机动画只更新 SpriteFrame，不修改服务器位置、网络快照坐标、朝向或生命条坐标转换。

## REQ-0.11 右下技能扇形布局

- 右下技能区域采用移动 MOBA 常见的扇形交互结构，但不复制具体游戏的美术资源。
- 普攻按钮作为右下技能区的主按钮和圆心，其它技能从普攻左侧沿圆弧向上排列。
- `ActionControlsRoot` 作为技能区域唯一整体容器，通过安全区 Widget 右下对齐；所有按钮内部只使用容器局部坐标。
- 技能区包含四个技能按钮：“鲫鱼抢食”占据最左下第一个圆弧槽位，其余三个技能依次沿圆弧向上排列。
- “鲫鱼抢食”沿用原冲刺撕咬的稳定技能 ID `skill-dash-bite`、图标、动画类型、服务端效果和冷却参数，改名不得破坏既有存档与多人协议兼容性。
- 普攻和技能名称显示在对应图标内部下侧，文字下边缘与图标下边缘对齐，并使用高对比描边保证可读性；禁止把名称放在图标外侧。
- 四个技能进入共享冷却时，图标上覆盖半透明深色径向蒙板；蒙板从 12 点方向开始顺时针转动一圈并随剩余冷却持续减少，冷却结束时完全消失。
- 冷却蒙板复用当前正式技能位图进行染色和径向裁剪，不新增 SVG、Graphics 或重复 PNG；技能名称显示在蒙板上层。
- 每个按钮的正式图像、触摸区域和文字标签必须属于同一个按钮节点，不得发生显示位置与事件位置分离。

## REQ-0.12 鲸吞技能

- 第二个圆弧技能槽位由临时技能改为正式技能“鲸吞”，临时采用 `ART-PROMPT-UI-0007 v1` 生成并登记的独立 PNG 图标；只替换该按钮的图片来源，不改变 `ActionControlsRoot`、按钮位置、尺寸、触摸区域、标签或冷却蒙板结构。该图标因含牙齿与原提示词排除项冲突，属于用户明确指定的暂用资源，后续应由已存档待确认的 v2 替换。
- 鲸吞由服务器在施法者 800 世界单位范围内选择最近的存活、非无敌敌方玩家；客户端不得提交或伪造目标。范围内没有有效目标时，服务器返回 `noTarget`，客户端取消本次冷却。
- 服务器确认目标后，将施法者的权威位置瞬移到目标当前权威位置，并向房间内所有客户端同步施法者、目标、位置和 3 秒效果时长。
- 鲸吞不造成额外伤害：施法者播放现有撕咬甩扯攻击动作，身体放大为正常尺寸的 3 倍并变为 50% 透明；被选中目标只变为 50% 透明。
- 3 秒后双方透明度恢复 100%，施法者恢复正常尺寸；目标的服务器坐标、生命值和动作状态不因该视觉效果改变。
- 鲸吞使用独立 8 秒冷却和独立径向蒙板；冲刺、临时技能 3、临时技能 4 继续共享原冲刺技能的 5 秒冷却。
- `skillEffect` 与动作状态快照携带 `targetId`、`effectDurationMs`/`actionRemainingMs`，保证晚加入或漏收瞬时事件的客户端能够补偿显示。

## REQ-0.13 配置驱动技能栏

- 玩家技能必须由单独的技能配置和技能栏配置装配；不得在 `GameBootstrap.ts` 中硬编码技能名称、图标路径、按钮槽位、圆弧角度、按钮尺寸、冷却时间、冷却组、攻击范围、冲刺距离、特效色值、特效时长或鲸吞缩放/透明度参数。
- 单个技能配置必须包含稳定 ID、网络技能 ID、UI 槽位、图标资源路径、冷却组和 `clientEffect`；技能栏配置必须包含右下整体容器布局和技能配置路径列表。
- `SkillActionPanel` 必须作为通用 UI 组件，传入配置与资源后创建按钮、标签、触摸区和冷却蒙板；只能通过激活回调连接玩法，不得直接修改服务器权威状态。
- `SkillEffectExecutor` 根据配置执行本地预测表现并发送技能意图；新增已有表现类型的技能不应改动 `GameBootstrap.ts` 的技能按钮创建代码。

## REQ-0.14 大世界主 UI 管理

- 大世界启动期间创建的固定屏幕 UI 必须由 `MainUIManager` 统一管理：`FishHealthOverlay`、操作提示、生命提示、`JoystickControlRoot` 和配置驱动的 `SkillActionPanel`。
- `GameBootstrap` 仅传入已加载资源、输入回调和技能激活回调，并持有管理器公开的节点/标签引用以协调网络、战斗和角色状态；不得在其中重新创建上述固定 HUD 节点。
- UI 仍固定在 `HudRoot/SafeAreaRoot/InputLayer`；角色、地图和特效仍只在 `WorldRoot`。本次职责拆分不得改变现有 1280×720 坐标语义、相机补偿或 Widget 约束。
## REQ-0.15 死亡翻滚技能

- 技能 3 使用稳定网络 ID `skill-death-roll`，服务器在一次释放中对命中目标结算一圈、扣除 3 点生命，并广播同一技能事件与伤害事件。
- 攻击者与被攻击者仅播放本地视觉翻滚和咬合表现；服务器位置、碰撞位置和移动状态不因视觉翻滚改变。
- 技能图标必须使用经过确认并存档的正式 PNG；当前代码在新图标确认前暂时复用冲刺图标，不得以 SVG 或 Graphics 代替正式资源。

## REQ-0.16 死亡翻滚旋转轴

- 死亡翻滚是纯表现动作，不改变服务器权威位置、朝向、命中和结算。
- 客户端表现必须绕角色局部 X 轴旋转一整圈；不得使用 2D `node.angle`（Z 轴）模拟翻滚。
- 鱼儿左右朝向仍由既有水平缩放配置控制，翻滚结束后恢复正常表现状态。
- 玩家根节点只负责世界位置；鱼 Sprite 位于独立 `FishVisual` 视觉子节点，翻滚仅修改该子节点局部 X 轴，避免被位置/朝向同步覆盖或误用根节点 2D Z 角度。

## REQ-0.17 大王喷墨

- 技能 4 稳定网络 ID 为 `skill-ink-splash`，名称为“大王喷墨”。
- 释放时记录服务器权威释放点；从该点向 16 个等角方向喷出 16 条长度 10 米的黑墨柱，喷洒表现持续 0.5 秒。
- 喷洒完成后等待 0.5 秒，再以原释放点为圆心快速扩散；扩散中心固定，不随释放者后续移动改变。
- 扩散结算时，原释放点 10 米半径内的其它存活玩家受到 25 点伤害；服务器负责范围、无敌、死亡和击杀结算，客户端只表现效果。
- 黑墨表现必须是由不规则墨团、散落墨滴和中心墨云组成的泼洒效果，不得绘制成长方形或胶囊形的连续墨柱。
- 技能冷却 10 秒；图标使用确认并登记的正式位图，不使用 SVG、Graphics 或临时图标替代按钮资源。

## REQ-0.18 技能装配与替换界面

- 主界面只保留普通攻击与 1—4 号四个主动技能槽位；后续新增技能进入全量技能库，不得继续扩张主界面按钮数量。
- `skill-library-player.json` 登记所有可用技能配置路径，`skill-loadout-player.json` 只定义默认装配与技能栏布局；运行时装配状态使用独立版本化本地记录。
- 主界面右上角提供固定屏幕的技能配置入口，入口与技能弹窗均属于 `HudRoot/SafeAreaRoot/InputLayer`，不得跟随地图或使用世界坐标。
- 点击入口打开技能配置界面。玩家先选择 1—4 号槽位，再从“未装备技能”列表选择技能完成替换；已经装备的技能不得重复占用其它槽位。
- 替换后立即刷新右下技能按钮的图标、名称、点击行为和冷却归属，不修改技能自身伤害、网络 ID 或服务器权威规则。
- 已装备技能 ID 使用存档格式版本 1 保存；存档缺失、损坏、重复或引用不存在技能时安全回退到默认四技能。
- 未装备技能支持分页显示；当前技能库没有未装备技能时必须显示明确空状态，后续只需向技能库加入配置即可自动出现。
- 右上入口图标必须使用经确认并登记的正式 PNG 美术资源；在 `ART-PROMPT-UI-0010 v1` 生成前仅允许复用现有已登记位图做功能占位，不得使用 SVG 或程序图形冒充正式图标。

## REQ-0.19 虎鲸冲刺

- 新增稳定技能 ID `skill-orca-charge`，显示名称为“虎鲸冲刺”；该技能只加入 `skill-library-player.json`，不得加入默认 `skill-loadout-player.json`，因此不会替换玩家当前主界面四技能。
- 玩家可在右上技能配置界面选择 1—4 号任意槽位，再从未装备技能列表选择“虎鲸冲刺”完成装配；被替换下来的技能回到未装备列表。
- 释放时由服务器在施法者前方 100°、900 世界单位范围内选择最近的存活、非无敌玩家；客户端不得提交目标或伪造目标坐标。
- 找到目标后，施法者迅速冲到目标原位置前方约 96 世界单位，目标沿冲刺方向被顶退 360 世界单位；双方最终坐标必须经过服务器地图边界限制。
- 虎鲸冲刺造成 60 点严重伤害，冷却 12 秒；伤害、死亡、经验和复活继续使用既有服务器权威战斗结算。
- 没有合法前方目标时服务器返回 `noTarget`，不消耗技能输入 tick 和冷却，客户端取消预测冷却并提示玩家。
- 服务器 `skillEffect` 同步施法者最终位置、目标 ID、目标最终位置和 650 毫秒表现时长；各客户端播放冲刺、攻击、受击和顶飞弧线表现。
- 顶飞期间目标根节点保持服务器权威最终位置，鱼的独立视觉子节点从旧位置沿短暂抬升弧线追赶到根节点，避免网络快照覆盖表现或改变命中坐标。
- 正式技能图标使用 `ART-PROMPT-UI-0011 v1`；确认并生成前临时引用已登记的冲刺技能 PNG，不复制资源、不使用 SVG 或程序化占位图标。

## REQ-0.20 角色变身与大王乌贼形态

- 主界面右上角新增“变身”入口，与“技能”入口放在同一个 `TopRightFeatureRoot` 内左右排列；整体固定在 `HudRoot/SafeAreaRoot/InputLayer`，不得跟随地图或分别使用屏幕坐标定位。
- 点击入口打开独立 `TransformDialog`。玩家可查看当前形象和已解锁形态，选择后立即更换主角形象；初始“小鲫鱼”和新增“大王乌贼”均由版本化形态配置登记。
- 当前形态 ID 使用独立存档格式版本 1 保存；存档缺失、损坏或引用不存在形态时安全回退到 `fish-player-crucian`。
- 变身只更换角色表现和动画资源，不改变玩家位置、朝向、生命、等级、经验、已装配技能、技能伤害或服务器战斗状态。
- 大王乌贼必须具备统一朝右的标准形象、6 帧循环游动、8 帧技能攻击和 8 帧无血腥受击资源；所有技能释放时使用当前形态的攻击动作，受伤时使用当前形态的受击动作，动作结束后恢复当前形态游动帧，不得退回鲫鱼贴图。
- 本地选择形态后向服务器提交稳定 `appearanceId`；服务器只允许白名单形态并在玩家快照中广播。新加入、断线重连和漏收瞬时事件时均通过快照恢复正确形态。
- 所有客户端必须为本地和远端玩家按 `appearanceId` 选择同一组游动、攻击和受击帧；形态资源朝向只读取配置，禁止在本地、远端或技能代码中分别写死翻转。
- 所有玩家形态的标准图、游动、攻击和受击资源必须由 `AnimationsResManager` 统一加载、缓存、方向归一和查询；`GameBootstrap` 只控制动作状态、播放时机、网络事件与角色节点，不得保存形态资源映射或成组动作帧数组。
- 变身入口图标和大王乌贼全部角色资源必须使用经用户确认的中文提示词生成并登记，不允许使用 SVG、程序化图形或现有技能图标临时拼凑。
